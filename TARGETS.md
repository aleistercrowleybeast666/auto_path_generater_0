取货区：

- `PICKUP_BOX_2` / `PICK_2`: `(1600, 0)`
- `PICKUP_BOX_3` / `PICK_3`: `(1855, -500)`

放货区：

- `DROP_BOX_4` / `F_DROP_4`: `(-1640, 875)`

保持场地边界、圆柱、对象尺寸、yaw、configured/enabled 和 physical site 映射不变。

## 1. 本轮目标

在当前真实 GUI 入口 `src/hjmb_pathgen/py_ui/v35_exact_main_window.py` 的“固定8点 / 最优路段 / 批量”页面加入一个显式的固定点避障检查入口。用户设置或调整 8 个逻辑固定点后，可以点击“检测固定点碰撞”，立即检查这些固定姿态是否与当前场地障碍物或边界发生碰撞。

本轮只完成固定点碰撞检查所需的前端和调用逻辑。后续会替换或调整避障模型，因此本轮不得修改现有碰撞模型、障碍物映射、车辆包络定义、连续碰撞算法、路径规划算法或 BIN 协议。

## 2. 固定范围

只检查以下 8 个逻辑点：

- `P_START`
- `P_PICK_1`
- `P_PICK_2L`
- `P_PICK_2R`
- `P_PICK_3`
- `P_DROP_1`
- `P_DROP_2`
- `P_DROP_3`

本功能是离散固定姿态检查，不检查固定点之间的连线、Bezier 曲线、Leg、完整 Case 或时间参数化轨迹，也不自动移动固定点或生成绕障路径。

## 3. 当前模型复用要求

- 复用现有 `build_collision_world(...)`、`check_pose_collision(...)`、`RobotPose` 和相关碰撞结果结构。
- 不在 UI 中重新实现几何碰撞算法。
- 新增一个纯服务层入口，建议文件为 `src/hjmb_pathgen/py_services/fixed_site_collision_service.py`。
- 服务层不得依赖 PySide6，不写文件，不修改 `ProjectV40`，不修改 Case。
- 服务层与 GUI 之间使用明确的检查报告对象或等价的结构化返回值，避免 GUI 解析自由文本。
- 当前服务边界应允许后续替换碰撞模型时只修改服务层，而不重做按钮和结果展示。

## 4. 待检测数据来源

点击按钮时必须检查当前界面中的尚未保存数据，而不是只读取磁盘上的旧 `project.json`。

应从当前 UI 状态构造一个只存在于内存中的 `ProjectV40` 快照，至少包含当前的：

- 8 个固定点位置和 yaw；
- 车辆 `r_large_mm`、`r_small_mm` 及现有碰撞配置；
- 场地边界和现有障碍物；
- 倒货姿态启用状态；
- 11 个倒货姿态的 `configured`、`yaw_ddeg`、`dx_mm`、`dy_mm`。

可以从 `_save_project_config()` 中提取“由当前 UI 构造 ProjectV40”的必要部分为一个小型辅助方法，再让保存和碰撞检查共同调用。只允许做这项直接相关的提取，不顺便重构其他保存流程；保存行为必须保持不变。

点击检测不得：

- 保存 `project.json` 或 Case；
- 启动路径生成或 worker；
- 改写 Case 的碰撞审核结果；
- 改变审批状态；
- 因检测本身把项目标记为 `STALE` 或 dirty。

## 5. yaw 与倒货姿态规则

### 5.1 普通固定点

当固定点 yaw 为具体值时，检查该点的一个 `RobotPose(x, y, yaw)`。

### 5.2 不约束 yaw

当固定点 yaw 为 `0xFFFF` 时，不得把它当成实际角度传给碰撞算法。

- 对 `P_DROP_1/2/3`：若已启用倒货姿态选择，则检查属于该逻辑放货站且 `configured=true` 的每一个倒货姿态。
- 每个倒货姿态的实际检测位姿为：
  - `x = fixed_site.x_mm + dx_mm`
  - `y = fixed_site.y_mm + dy_mm`
  - `yaw = yaw_ddeg`
- 倒货姿态与逻辑放货站的对应关系使用现有 `EXPECTED_UNLOAD_POSE_CATALOG[profile_id]["station_site"]`，不要在新服务中再维护一份独立映射表。
- `configured=false` 的姿态不参与检查。
- 若某放货站没有任何已配置姿态，则该站结果为“未完整检测”，不能判为通过。
- 对其他 yaw 为 `0xFFFF` 的固定点，结果为“未完整检测：未指定 yaw”，不能擅自取 0°、上一点 yaw 或其他默认值。

若未启用倒货姿态选择，但放货固定点本身具有具体 yaw，则按一个普通固定姿态检查；若仍为 `0xFFFF`，则按未完整检测处理。

## 6. 结果判定

总结果只使用以下三类语义：

- `PASSED`：所有要求检查的固定姿态都完成检查且没有碰撞。
- `FAILED`：至少一个固定姿态发生穿透碰撞。
- `INCOMPLETE`：没有发现碰撞，但至少一个点因未配置、yaw 不明确、无可用倒货姿态或配置错误而未完成检查。

如同时存在碰撞和未完整项，总结果为 `FAILED`，同时在详情中保留未完整项。

`TOUCHING` 沿用当前碰撞模型语义，仍视为合法；只有 `PENETRATING` 判定为碰撞。

每个已检查姿态至少保留：

- 固定点 key；
- 倒货姿态 profile id（普通固定点可为空）；
- 实际 `x/y/yaw`；
- 是否通过；
- 最小有符号间隙；
- 最近障碍物 ID；
- 碰撞时的障碍物 ID、类型和穿透深度。

## 7. GUI 要求

在“固定8点 / 最优路段 / 批量”页的固定点表附近增加：

- 按钮：`检测固定点碰撞`
- 简短状态标签，初始显示：`固定点碰撞：未检测`

点击后同步完成检查即可；最多只有 8 个固定点和少量倒货姿态，不需要新增 worker。

检查完成后：

- 状态标签显示 `通过`、`发现碰撞` 或 `检测不完整`，并给出通过/碰撞/未完整数量；
- 弹出一个结果对话框，按固定点列出检查详情；
- 详情至少显示最小间隙和最近障碍物；发生碰撞时显示碰撞障碍物和穿透深度；
- 对使用多个倒货姿态的放货点，按 profile id 分行显示，不得只显示聚合结果；
- 无项目、配置无法构造或碰撞世界无法建立时，给出明确错误，不崩溃。

本轮不要求在场地图上高亮碰撞包络，不新增固定点表列，不保存检查报告。

## 9. 最小测试

新增或修改测试只覆盖本功能：

1. 服务层：8 点全部安全时为 `PASSED`。
2. 服务层：至少一个固定点穿透障碍物时为 `FAILED`，并返回障碍物详情。
3. 服务层：放货点使用已配置 profile 的 `dx/dy/yaw` 检查。
4. 服务层：`0xFFFF` 且无可用实际 yaw 时为 `INCOMPLETE`，不得按 0° 检查。
5. GUI：按钮和初始状态标签存在。
6. GUI：点击按钮使用当前未保存的固定点数据，且不调用保存、规划或 worker。
7. GUI：结果标签能区分通过、碰撞和不完整。

只运行新增服务测试和直接相关的 GUI 测试，不运行全量回归。
