# HJMB 物流赛比赛场地几何信息

## 1. 文件用途

本文件是路径文件生成工程中比赛场地几何的权威说明，供 Codex 修改生产默认场地和当前 `project.json` 时使用。

依据：

- 《中国大学生机械工程创新创意大赛竞赛公告》第 7 页图 3；
- 工程坐标系现有定义；
- 用户对放货区尺寸标注和中心坐标的确认。

本轮只修正场地对象的位置，并让固定 8 点碰撞检测结果弹窗支持滚动查看。不得修改碰撞算法、车辆包络、障碍物类型映射、连续碰撞检测、路径规划或 BIN 协议。

## 2. 坐标系

工程继续使用当前坐标系，不得改变：

- 原点：场地中心；
- 场地长边为 x 轴，范围 `[-2000, 2000] mm`；
- 场地短边为 y 轴，范围 `[-1000, 1000] mm`；
- `+x` 指向取货区；
- `-x` 指向放货区；
- `+y` 对应图 3 上方；
- `-y` 对应图 3 下方；
- yaw 零方向仍为机器人正面朝取货区。

名义可用场地尺寸为 `4000 mm × 2000 mm`。图中的 `4200 mm × 2200 mm` 是包含外围结构的尺寸，不得替换碰撞边界。

### 3.4 放货区五个货箱

#### 3.4.1 图纸尺寸与碰撞对象尺寸的区别

图 3 在放货区标出的 `300 mm × 210 mm` 是用于确定放置位置中心的矩形尺寸。靠场地最近边还有 `20 mm` 间隙，因此：

- 对 5、6、7 号位，中心距 `-x` 边界为 `210 / 2 + 20 = 125 mm`；
- 对 4、8 号位，中心距最近的 `±y` 边界也为 `210 / 2 + 20 = 125 mm`。

这里的 `300 × 210` 只用于读取图纸并计算中心位置，**不得据此把碰撞障碍物尺寸改成 300 × 210**。

放货箱碰撞对象继续使用工程当前基于货箱实际外形设置的尺寸：

- 4、8 号箱：`length_mm=280`，`width_mm=200`；
- 5、6、7 号箱：`length_mm=200`，`width_mm=280`。

本轮只修改中心坐标，不修改尺寸、yaw、ID、启用状态或物理站点映射。

#### 3.4.2 正确中心坐标

| obstacle_id | physical_drop_site | center_x_mm | center_y_mm | length_mm | width_mm | yaw_ddeg |
|---|---|---:|---:|---:|---:|---:|
| `DROP_BOX_4` | `F_DROP_4` | -1640 | 875 | 280 | 200 | 0 |
| `DROP_BOX_5` | `F_DROP_5` | -1875 | 400 | 200 | 280 | 0 |
| `DROP_BOX_6` | `F_DROP_6` | -1875 | 0 | 200 | 280 | 0 |
| `DROP_BOX_7` | `F_DROP_7` | -1875 | -400 | 200 | 280 | 0 |
| `DROP_BOX_8` | `F_DROP_8` | -1640 | -875 | 280 | 200 | 0 |

位置依据：

- 5、6、7 号位的 y 坐标保持 `400/0/-400 mm`；
- 5、6、7 号位中心距 `-x` 场地边界为 `125 mm`，所以 x 为 `-2000 + 125 = -1875 mm`；
- 4、8 号位中心距 `-x` 场地边界仍为 `360 mm`，所以 x 为 `-2000 + 360 = -1640 mm`；
- 4 号位中心距 `+y` 边界为 `125 mm`，所以 y 为 `1000 - 125 = 875 mm`；
- 8 号位中心距 `-y` 边界为 `125 mm`，所以 y 为 `-1000 + 125 = -875 mm`。

当前工程旧值及最终修正如下：

| 对象 | 旧中心 | 最终正确中心 |
|---|---|---|
| `DROP_BOX_4` | `(-1500, 800)` | `(-1640, 875)` |
| `DROP_BOX_5` | `(-1700, 400)` | `(-1875, 400)` |
| `DROP_BOX_6` | `(-1700, 0)` | `(-1875, 0)` |
| `DROP_BOX_7` | `(-1700, -400)` | `(-1875, -400)` |
| `DROP_BOX_8` | `(-1500, -800)` | `(-1640, -875)` |

不得使用上一版文档中的 `±880` 或 `-1880`。

## 5. 生产数据修改范围

必须同步修改：

1. `src/hjmb_pathgen/py_services/project_bootstrap_service.py` 中新建 V4 工程的生产默认 `field_objects`；
2. 仓库当前 `project.json` 中的 `field_objects`，只改本文件列出的放货箱中心坐标；
3. 与 V4 workspace bootstrap 直接相关的最小测试，明确断言上述生产默认坐标。

取货区已经正确，本轮不要再修改取货区对象。

不得借本轮修改：

- `sites` 中已经标定的 8 个逻辑固定点；
- `unload_pose_profiles`、`unload_profiles`；
- 车辆大/小包络半径；
- `FOOTPRINT_BY_OBSTACLE_TYPE`；
- 障碍物跳过规则；
- 圆柱、边界、货箱尺寸和 yaw；
- `example_project_service.py` 的演示性场景；
- 合成测试夹具中为了单元测试构造的任意场景；
- 已生成的 Case、BIN、Leg 库或发布目录。

场地坐标变化会改变碰撞配置 hash。已有路径应按现有 STALE/重新验证机制处理，不得手工篡改旧结果使其继续显示为通过。

## 6. GUI 显示要求

场地图继续从当前项目的 `field_objects` 绘制，不新增第二套硬编码坐标。修正 `project.json` 后，GUI、固定 8 点碰撞检测、路径碰撞检测和规划器必须读取同一组场地对象。

固定 8 点碰撞检测的详情弹窗必须可滚动：

- 使用可调整大小的 `QDialog`；
- 详情区使用 `QTextBrowser`、`QPlainTextEdit(readOnly)` 或 `QScrollArea`；
- 碰撞详情较多时出现垂直滚动条，不截断后续固定点或倒货姿态；
- 保留现有检查结果、状态标签和判定逻辑，不改碰撞模型；
- 关闭弹窗后不改变项目 dirty/STALE 状态，不产生文件写入。

## 7. 验收条件

- 新建工程和当前仓库 `project.json` 均使用第 3.4.2 节的放货区中心坐标；
- `DROP_BOX_4/8` 为 `(-1640, ±875)`；
- `DROP_BOX_5/6/7` 的 x 均为 `-1875`，y 分别为 `400/0/-400`；
- 不得出现上一版错误坐标 `(-1640, ±880)` 或 `(-1880, y)`；
- 取货区、两个圆柱、边界、对象尺寸和 yaw 未被误改；
- `300 × 210` 只用于图纸中心位置计算，没有改写碰撞对象尺寸；
- 场地图和所有碰撞入口使用同一份 `field_objects`；
- 固定 8 点碰撞详情弹窗可滚动并能查看全部内容；
- 只运行与 bootstrap 默认场地和该弹窗直接相关的最小测试。
